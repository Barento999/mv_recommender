💾 STORAGE OPTIONS
═══════════════════════════════════════════════════════════════════════════════

OPTION 1: In Memory (Default) ✅
  Location: RAM (global variables in pipeline.py)
  When: On app startup
  Speed: ⚡ Very fast (50-100ms inference)
  Persistent: ❌ No (lost on restart)
  Storage: ~150-200 MB RAM
  Perfect For: Production API
  Code: app/ml/pipeline.py lines 20-24

OPTION 2: On Disk (Optional)
  Location: backend/models/ directory
  When: Only when you run train_complete_project.py
  Speed: 🐢 Slower (hundreds of ms to load)
  Persistent: ✅ Yes (survives restart)
  Storage: ~5-7 MB disk
  Perfect For: Backup, archive, multi-instance
  Code: app/ml/ml_manager.py (save_models function)

🔄 REQUEST FLOW (How Models Are Used)
═══════════════════════════════════════════════════════════════════════════════

User makes request:
  GET /recommendations?limit=10

Flow:
  1. app/routes/recommendations.py receives request
  2. Calls recommendation_service.get_recommendations()
  3. Calls pipeline.get_recommendation(user_id)
  4. Pipeline checks _global_cache (TTL + LRU)
     
     Cache HIT (60-80% of time):
       ✅ Return cached result in 20-30ms

     Cache MISS (20-40% of time):
       → Use _global_cf_model to predict
       → Takes 50-100ms inference
       → Cache the result
       → Return to user

Total response time: 20-100ms
Cache hit rate: 60-80%

💡 KEY CONCEPTS
═══════════════════════════════════════════════════════════════════════════════

Global Variables:
  Models are stored as global variables in pipeline.py
  Once set, they persist for the entire app lifetime
  Shared across all requests (thread-safe access)
  No disk I/O needed

Singleton Pattern:
  Only one instance of each model
  Created on startup, reused forever
  Perfect for API servers
  Minimizes memory usage

Getter Functions:
  get_cf_model()          → Returns _global_cf_model
  get_content_model()     → Returns _global_content_model
  get_cache()             → Returns _global_cache
  get_pipeline_status()   → Returns health metrics

❓ FAQ
═══════════════════════════════════════════════════════════════════════════════

Q: Do I need to save models to disk?
A: No, not for production. Retrain on startup takes only 3-7 seconds.

Q: Can I save models to disk?
A: Yes, run: python train_complete_project.py
   This saves to backend/models/ directory.

Q: Are models lost when I restart the app?
A: Yes, they're retrained on next startup (3-7 seconds).
   This is intentional - ensures fresh training.

Q: How do models persist across requests?
A: Global variables in pipeline.py stay in memory for the app lifetime.
   All requests use the same global instances.

Q: Can I load pre-trained models from disk?
A: Yes, modify pipeline.py to load from backend/models/ instead of training.
   Advanced - requires code changes.

Q: Where is model data stored if not on disk?
A: In RAM as Python objects (NumPy arrays, scikit-learn models, etc).
   Total: ~150-200 MB.

Q: How do I know models are loaded?
A: Check: curl http://localhost:8000/recommendations/status
   If "collaborative_filtering": true, models are loaded.

Q: Can multiple instances share models?
A: No, each instance trains its own. For sharing, use disk storage.
